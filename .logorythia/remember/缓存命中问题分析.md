# 缓存命中问题分析

## 核心问题：OpenAI 流式链路未设置 `include_usage`

**文件引用**：`file:///Volumes/知阳/开发/Packges/SwiftOpenAI/Sources/SwiftOpenAI/ProviderSupport.swift:<569>-<586>`

`applyProviderDefaults` 中 `stream_options.include_usage` 仅对 Minimax 开启。OpenAI Chat Completions API 的流式响应默认不返回 `usage`（含 `cached_tokens`），必须显式设置 `stream_options: {"include_usage": true}` 才会在最后一个 chunk 附带。

**后果**：流式调用（`sendMessage` 默认 `stream: true`）时，`cachedTokens` 永久为 nil。非流式路径（`sendMessageSync`）不受影响。

**数据流验证**：`file:///Volumes/知阳/开发/Packges/SwiftOpenAI/Sources/SwiftOpenAI/Streaming/StreamingSupport.swift:<262>-<264>` — 由于服务端未返回 usage，`setUsage` 永不触发。

## Codex/Responses 路径 `store: false` 硬编码

**文件引用**：`file:///Volumes/知阳/开发/Packges/SwiftOpenAI/Sources/SwiftOpenAI/Streaming/ResponsesSupport.swift:<216>-<223>`
Windows 分支：`file:///Volumes/知阳/开发/Packges/SwiftOpenAI/Sources/SwiftOpenAI/Streaming/ResponsesSupport+Windows.swift:<298>`

`store: false` 表示不持久化对话。Responses API 以「对话」为粒度管理状态，不存储意味着后续多轮无法通过 `previous_response_id` 延续上下文，削弱服务端侧基于会话的前缀复用。

## `AnyCodableValue` 数字精度回退

**文件引用**：`file:///Volumes/知阳/开发/Packges/SwiftOpenAI/Sources/SwiftOpenAI/Models/ChatQuery.swift:<463>-<483>`

`Int` → `Double` 的 decode 策略可能导致请求体字节级差异：当上游 JSON 数字以 `42.0`（浮点字面量）出现时，落入 `Double` 分支，序列化为 `42.0` 而非 `42`。对于依赖 exact-match 的前缀缓存构成潜在 miss。

## 流式 usage setter 为替换语义

**文件引用**：`file:///Volumes/知阳/开发/Packges/SwiftOpenAI/Sources/SwiftOpenAI/Streaming/OpenAISendMessageValueHelper.swift:<113>-<115>`

`setUsage` 完全替换而非合并。当前主流厂商只在最后一个 chunk 返回 usage，语义正确。但若某厂商分多 chunk 携带递增 usage（先发 `prompt_tokens`，后发 `completion_tokens` 与 `cached_tokens`），后到 chunk 会覆盖前值，导致 `cachedTokens` 清零。目前无已知触发。

## DeepSeek 缓存命中率约 1% 的根因分析

**1% 极低命中率，SDK 层面已确定无引入非确定性字段。** 已修复的三项（`include_usage` 全覆盖、移除 `store: false`、`AnyCodableValue` 整型归一化）均增强而非削弱请求确定性。

**DeepSeek 的 Prompt Cache 按前缀精确匹配工作**，缓存键含：model、messages 前缀、temperature、top_p、presence_penalty、frequency_penalty、stop、max_tokens。1% 意味着几乎所有请求之间这些要素都有差异。常见情形：

1. **每次都是新对话**：系统提示词不同或首条用户消息不同，前缀无重叠。
2. **多轮对话中前缀占比低**：长对话（如 8000 token）仅系统提示词 200 token 命中，命中率被稀释（~2.5%）。Reasoner 模型（R1）思考过程 token 进一步稀释。
3. **采样参数变化**：temperature/top_p 每次不同视为不同缓存键。
4. **`extraBody` 变化**：传入会变的字段（时间戳、随机种子）导致每次请求体不同。
5. **缓存 TTL 过短**：DeepSeek prompt cache TTL 通常 5–30 分钟，请求间隔超出则过期。

**排查方法**：抓取相邻两次请求的原始 request body 逐字节比对。`ProviderRequestEncoder.makeRequest` 返回后将 `urlRequest.httpBody` 写入日志，用 `diff` 对比。JSON 一致则问题在前缀占比或 TTL；不一致则定位差异字段。
